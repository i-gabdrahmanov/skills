# GigaCode Migrator

Переносит дерево скиллов `~/.gigacode/skills` в GigaCode-вариант `~/.gigacode/skills`,
заменяя бренд, пути и CLI-вызовы по редактируемой таблице правил. Чистый
ребрендинг — никакой логики скиллов не меняет.

## Что меняет (и чего НЕ меняет)

Правила в `references/mapping.json`, применяются **по порядку сверху вниз** —
специфичные раньше общего, чтобы общий `gigacode→gigacode` не испортил ссылки и команды:

| Категория | Из | В |
|---|---|---|
| url | `gigacode.ru`, `gigacode.ru`, `gigacode.ru`, `gigacode.ru` | `gigacode.ru` |
| cli | `gigacode -p` | `gigacode -p` |
| product | `GigaCode`, `GigaCode` | `GigaCode` |
| dotpath | `.gigacode` | `.gigacode` |
| brand_generic | `gigacode` (любой регистр) | `gigacode` (регистр сохраняется: `Gigacode`→`Gigacode`, `GIGACODE`→`GIGACODE`) |

**Модели НЕ мапятся специально.** Скиллы не выбирают модель — ни в одном `SKILL.md`
нет поля `model:`. Модель задаёт рантайм GigaCode (под капотом Qwen). Единственное
место с моделями — тулинг `skill-creator`, где флаг `--model` по умолчанию берёт
модель текущей сессии. Поэтому раздел `model_id` из таблицы убран осознанно.

> Косметика: единичный пример `gigacode-sonnet-4-20250514` в `skill-creator/references/schemas.md`
> общее правило превратит в `gigacode-sonnet-4-20250514` (бессмысленно, но безвредно —
> это иллюстрация в доке, не вызов). Если мешает — поправь вручную после миграции.

## Как запускать

Скрипт: `scripts/migrate.py`. По умолчанию `--source ~/.gigacode/skills`,
`--target ~/.gigacode/skills`, недеструктивно (копия).

### Шаг 1 — ВСЕГДА сначала dry-run

```bash
python ~/.gigacode/skills/gigacode-migrator/scripts/migrate.py --dry-run
```

Выдаёт JSON-сводку: сколько файлов затронуто, сколько замен, какие
переименования, и `sample` из первых файлов. Ничего не пишет. Покажи сводку
пользователю.

### Шаг 2 — реальная миграция в копию (рекомендуемо)

```bash
python ~/.gigacode/skills/gigacode-migrator/scripts/migrate.py
# повторный прогон поверх существующего target:
python ~/.gigacode/skills/gigacode-migrator/scripts/migrate.py --force
```

Копирует дерево в `~/.gigacode/skills` и правит копию. `~/.gigacode` остаётся как есть.

### Опции

| Флаг | Назначение |
|---|---|
| `--dry-run` | только сводка, без записи |
| `--source PATH` | что мигрировать (можно весь `~/.gigacode`, не только `skills`) |
| `--target PATH` | куда положить результат |
| `--in-place` | **деструктивно**: править source напрямую. Требует явного согласия пользователя |
| `--force` | перезаписать существующий target |
| `--no-rename` | не переименовывать файлы/папки, только содержимое |
| `--mapping PATH` | своя таблица правил |

## Правила работы (для агента)

1. **Сначала dry-run, всегда.** Показывай сводку до любой записи.
2. **`--in-place` — только с явного «да».** Репозиторий скиллов не под git;
   правка на месте необратима. По умолчанию — копия.
3. **GigaCode = перелицованный Qwen.** Если пользователь захочет принудительно
   прописать конкретные Qwen-модели где-то в тулинге — это отдельная правка
   таблицы (`references/mapping.json`), не часть базового ребрендинга.
4. **Таблица — единственный источник правды.** Новые замены добавляй в
   `mapping.json`, а не в код. Порядок правил важен: специфичные выше общего.
5. **Проверка после миграции:** `grep -ril "gigacode" ~/.gigacode/skills` должен
   вернуть только осознанные остатки (если есть). Сообщи о них пользователю.

## Что НЕ делает

- Не запускает мигрированные скиллы и не проверяет их работоспособность под
  GigaCode-рантаймом — только текстовый ребрендинг.
- Не настраивает сам рантайм GigaCode (эндпойнт, модели, авторизацию).
- Не трогает бинарные файлы (картинки, jar, pdf, pptx) — копирует как есть.
- Не удаляет оригинал `~/.gigacode` (если только не `--in-place`).
