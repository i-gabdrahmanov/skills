# Jira через Atlassian MCP

Этот файл описывает, что именно нужно получить из Jira и в каком виде. Точные имена
инструментов зависят от установленного MCP-сервера — ищи их по шаблонам в основном файле
скилла.

## Получение задачи

При получении issue запрашивай (если MCP позволяет фильтровать поля):

- `key`, `summary`, `description`
- `issuetype.name`, `priority.name`, `status.name`
- `assignee.displayName`, `reporter.displayName`
- `labels`, `components[].name`
- `comment.comments[]` — последние 5-10, с `author.displayName`, `body`, `created`
- `attachment[].filename`

### Что значимо для анализа

| Поле | Зачем |
|---|---|
| `summary` + `description` | Что сломано, ожидаемое поведение |
| `comment` | Часто там лежит уточнение от QA/тимлида — оно важнее, чем оригинал |
| `attachment` | Скриншоты/логи. Имена дают подсказку (`stacktrace.txt`, `error.png`) |
| `components`, `labels` | Подсказка, какую часть кода править |
| `priority`, `issuetype` | Срабатывает скоуп-чек |

Описание в Jira может быть в ADF (Atlassian Document Format) или wiki-markup. Если ADF —
извлеки текст из `content[].content[].text`. Если плохо парсится — покажи "как есть"
пользователю и попроси кратко передать суть.

## Добавление комментария-отчёта

- Формат тела: Markdown, если MCP принимает; иначе обычный текст (большинство Jira-MCP
  принимают plain/markdown и сами конвертируют).
- Не меняй статус задачи автоматически. Если пользователь хочет — он сам попросит.
- После успешной отправки покажи пользователю короткое подтверждение: "Комментарий
  отправлен в STOR-123" с ссылкой, если MCP вернул её.

## Ошибки

- `403 Forbidden` / `401` — проблема с токеном. Не пытайся обходить, сообщи пользователю.
- `404 Not Found` — неверный ключ задачи. Проверь регистр и формат, спроси у
  пользователя.
- Если MCP вернул объект ошибки без HTTP-кода — покажи `error.message` пользователю.
