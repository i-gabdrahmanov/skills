# Создание задач в Jira через MCP

Точные имена инструментов зависят от установленного MCP-сервера (официальный Atlassian
MCP, сторонний `mcp-atlassian` и т.п.). **Не угадывай — найди подходящий инструмент в
списке доступных** и сверься с его схемой параметров.

## Карта действий → инструменты

| Действие | Что искать в доступных инструментах |
|---|---|
| Узнать типы задач проекта | `*issue*types*`, `*project*meta*`, `*createmeta*` |
| Создать issue | `*create*issue*`, `*jira*create*` |
| Создать подзадачу | тот же create-issue, но с типом подзадачи + полем `parent` |
| Связать задачи (blocks/relates) | `*create*link*`, `*issue*link*` |
| Назначить исполнителя / метки | поля `assignee`, `labels` в create, или `*update*issue*` |
| Получить URL задачи | обычно в ответе create (`self`/`key`); URL = `<base>/browse/<key>` |

## Поля при создании

Минимальный набор для create-issue (имена полей зависят от сервера, частые варианты):

- `project` / `projectKey` — ключ проекта (`STOR`).
- `issuetype` / `issueTypeName` — имя типа из шага «определи схему».
- `summary` — заголовок.
- `description` — тело. Уточни формат: некоторые серверы принимают plain text, другие
  Atlassian Document Format (ADF) или wiki-markup. Если непонятно — начни с простого
  текста; критерии приёмки оформляй маркерами `- [ ]` или просто списком.
- `parent` — для Sub-task: ключ родительской Story (`{"key": "STOR-200"}` или строкой,
  зависит от схемы).
- `labels`, `assignee` — если заданы конфигом.

## Различия схем (на что смотреть)

1. **Есть ли подзадачи.** Часть проектов отключает Sub-task. Если тип подзадачи не
   найден среди доступных — используй обычную Task и связь «relates to» / «is part of»
   со Story. Обязательно отметь это в черновике на гейте.

2. **Story vs Task vs История.** Тип верхнего уровня может называться по-разному. Бери
   первый подходящий: Story → История → Task. Не падай, если «Story» нет.

3. **parent для подзадач.** У подзадачи родитель обязателен и задаётся при создании
   (изменить тип на Sub-task после создания часто нельзя). Поэтому сначала Story, потом
   подзадачи с её ключом.

4. **Связи зависимостей.** «blocks»/«is blocked by» создаются отдельным link-вызовом
   после создания обеих задач, не полем create. Если link-инструмента нет — отрази
   зависимость текстом в описании.

## Порядок вызовов (надёжный)

1. createmeta → определить типы.
2. create Story → запомнить `key`.
3. для каждой задачи (в порядке `depends_on`): create Sub-task с `parent = story key`.
4. для каждой пары зависимостей: create link «is blocked by».
5. собрать `jira-tasks-result.json` с маппингом `task_id → key` и URL.

На любой ошибке после шага 2 — верни уже созданное (Story + успевшие подзадачи), не
пытайся «доделать любой ценой».
