# SDD — Specification-Driven Development документ (PDLC v3.5)

В v3.5 **спецификация первична, код производен** (стр. 66). SDD-документ устроен так, чтобы агент
получил его и реализовал задачу **без дополнительных вопросов**. tech-design генерирует `sdd.md`
рядом с `tech-design.md` в папке фичи и проставляет `sdd_ref` в каждую задачу task-plan.

## Обязательные разделы `sdd.md`

```markdown
# SDD: <фича>

## 1. Бизнес-контекст
Зачем фича, какую проблему решает, кто пользователь. (2–5 предложений, из BRD.)

## 2. Функциональные требования (Given-When-Then)
По каждому сценарию — приёмочный критерий в формате:
- **Given** <предусловие> **When** <действие> **Then** <ожидаемый результат>
Покрыть и happy-path, и ошибочные ветки.

## 3. Нефункциональные требования (NFR)
Производительность, нагрузка, безопасность, идемпотентность, лимиты, SLA — что применимо.

## 4. API-контракты
Для каждого нового/изменённого эндпойнта: метод, путь, request/response схема, коды ошибок.
Для Kafka: топик, ключ, схема сообщения, направление (producer/consumer).

## 5. Модель данных
Затрагиваемые entity/таблицы, новые поля, миграции (Liquibase changeset).

## 6. Критерии приёмки (Acceptance)
Сводный чек-лист «сделано», по которому собирается evidence bundle. Каждой задаче task-plan
соответствует ≥1 пункт Given-When-Then.

## 7. Риски и допущения
Что может пойти не так, какие предположения сделаны.
```

## Привязка к task-plan.json

Каждая задача обязана иметь:
- `acceptance`: массив строк Given-When-Then (≥1), копируется/ссылается из раздела 2/6 SDD;
- `sdd_ref`: якорь на раздел SDD, напр. `"docs/feature-pipeline/<slug>/sdd.md#t1"`.

Гейт `check_sdd.py` проверяет наличие `sdd.md` (с обязательными секциями) и `acceptance`+`sdd_ref`
у каждой задачи; без этого фаза дизайна (02-design) не закрывается, а evidence bundle неполон.

## Зачем строго

Строгий SDD = меньше «додумывания» агентом, выше autonomy-уровень (агент реализует без вопросов),
и прямая трассировка требование → задача → тест → evidence. Это ядро Intent Loop (стр. 59).
