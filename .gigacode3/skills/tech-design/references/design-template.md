# Шаблон tech-design.md

Держись этой структуры: разделы и заголовки фиксированы, чтобы оркестратор
`feature-pipeline` и разработчик `java-spring-dev` находили нужное предсказуемо.
Пропускать разделы можно только явной пометкой «не применимо» — это сигнал, что вопрос
рассмотрен, а не забыт.

````markdown
# Tech Design: <Название фичи>

**Jira:** <KEY, напр. KIDPPRB-8639 — ПЕРВОЙ строкой, если задача из Jira; иначе опусти строку>
**BRD:** ./brd.md
**Дата:** <YYYY-MM-DD>
**Статус:** Черновик

## 1. Резюме решения
2-4 предложения: что строим технически и каким образом. Без воды — суть подхода.

## 2. Затронутые модули
Из system-analysis. Какие модули меняем, какие только читаем. Если проект
многомодульный — укажи модуль для каждого изменения.

## 3. Изменения по слоям
Таблица в терминах java-spring-dev. Для каждого артефакта: новый или изменяемый,
и зачем. Это ядро дизайна — по нему пишется код.

| Слой | Класс | Новый? | Назначение |
|---|---|---|---|
| migration | `2026-05-add-export-job.xml` | новый | таблица export_job |
| entity | `ExportJob` | новый | задача экспорта |
| repository | `ExportJobRepository` | новый | CRUD + поиск по статусу |
| dto | `ExportRequest` | новый | тело POST |
| dto | `ExportResponse` | новый | ответ (id + статус) |
| mapper | `ExportMapper` | новый | ExportJob → ExportResponse |
| service | `ExportService` (+Impl) | новый | запуск/чтение экспорта |
| controller | `ExportController` | изм. | + POST /exports, GET /exports/{id} |

## 4. Миграции БД
Liquibase changeset (НЕ Flyway). Формат и нейминг — по конвенции существующего
`db/changelog`. Для каждого changeset: какие таблицы/колонки/типы/индексы/FK.
Если доменная модель не меняется — «Не требуется».

## 5. Async и интеграции
Kafka-топики (новые события, продюсеры/консьюмеры), внешние клиенты (Feign/WebClient).
Если ничего не затрагивается — «Не затрагивается» (явно, чтобы не гадали).

## 6. Контракт API
Для каждого эндпойнта: метод, путь (в стиле существующих), request/response DTO,
коды ошибок, требования авторизации.

| Метод | Путь | Request | Response | Коды | Auth |
|---|---|---|---|---|---|
| POST | /api/v1/exports | ExportRequest | ExportResponse | 202, 400, 401 | да |
| GET | /api/v1/exports/{id} | — | ExportResponse | 200, 404, 401 | да |

## 7. Влияние на существующее
Что может сломаться. Обратная совместимость (меняются ли публичные сигнатуры,
форматы ответов, схема). Какие существующие тесты затрагиваются.

## 8. Риски и альтернативы
- **Риски:** технические/данные/производительность.
- **Альтернативы:** 1-2 отвергнутых подхода с причиной отказа.
- **Допущения:** что взято из BRD или предположено (мелкие развилки, не дошедшие до
  вопроса пользователю).

## 9. Критерии приёмки
Из BRD §12, уточнённые технически. Это то, что должно стать зелёным.
````

## Принципы заполнения

- **Раздел 3 — главный.** Если он пуст или расплывчат, весь план бесполезен. Каждая
  строка должна давать разработчику однозначное «создать класс X со смыслом Y».
- **«Не требуется»/«Не затрагивается» — это контент**, а не пропуск. Он говорит
  разработчику: вопрос рассмотрен, тут пусто намеренно.
- **Согласованность с task-plan.json.** Каждая задача в плане должна покрывать какие-то
  строки раздела 3. Не должно быть изменений в разделе 3, не попавших ни в одну задачу.
