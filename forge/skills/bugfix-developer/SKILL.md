# Bugfix Developer

Скилл реализует минорный фикс в существующем Java/Spring Boot коде. Главный агент сам
правит файлы через `edit`/`write_file` — этот скилл задаёт **принципы и чек-лист**, чтобы
правка получилась минимальной, безопасной и не уехала в рефакторинг.

**Отличие от java-spring-dev:** тот — про *как писать новый код*. Этот — про *как
менять уже работающий код с минимумом риска*. Конвенции (Lombok, layer-first и т.д.)
бери из java-spring-dev — при необходимости вызови его параллельно. Принципы изменения
существующего кода — здесь.

---

## Принципы багфикса

### 1. Минимальное изменение

Найди самую короткую правку, решающую конкретный сценарий из задачи. Если три строки
в одном методе закрывают баг — этого достаточно.

❌ "По пути отрефакторил соседний метод, он же тоже коряво написан"
✅ "Изменил три строки в `Foo.bar()`, остальное не трогал"

Если в процессе нашёл рядом ещё проблему — не правь. Зафиксируй её отдельной репликой
пользователю: *"кстати, ещё нашёл X — оформить отдельной задачей?"*.

### 2. Сохраняй публичные сигнатуры

Не меняй без явного запроса:
- сигнатуры публичных методов (имена, типы аргументов, возвращаемый тип),
- имена/типы публичных полей,
- exception типы в `throws`,
- URL в `@RequestMapping`/`@GetMapping`/...,
- имена бинов в `@Bean`/`@Component(name=...)`.

Если сигнатуру **действительно** нужно поменять — СТОП. Сообщи пользователю и спроси
прежде чем делать. Тихие изменения публичного API в багфиксе — главный источник
регрессий в проде.

### 3. Не вводи новых абстракций

Багфикс — не повод создать новый интерфейс, сервис, конфиг-класс, утилитный метод.

❌ "Создал `ValidationStrategy` чтобы было гибче на будущее"
✅ "Добавил один `if` в существующий метод"

Если фикс действительно требует нового класса (например, новое исключение) — обсуди с
пользователем отдельно.

### 4. Стиль файла важнее своих привычек

Открой файл, который правишь. Если там:
- guard clauses → используй guard clauses, а не nested if/else.
- early return → не оборачивай в один big return в конце.
- `@Slf4j` → не вводи `Logger` вручную.
- builder-паттерн для DTO → не пиши `new Foo(); foo.setX(...)` рядом.

**Соответствие соседнему коду > соответствие "идеальному" паттерну.**

Если непонятно — посмотри ещё пару методов в этом же файле, увидишь паттерн.

### 5. Edge cases фиксим сразу

Если фикс добавляет проверку на null — подумай про:
- пустую коллекцию (`isEmpty()`, не только `null`),
- пустую/blank строку (`isBlank()`),
- значение по умолчанию,
- регистр строк (если сравниваешь как `equals`).

"Сделаю минимально, потом дополним" обычно превращается в "сломалось в проде через
месяц".

### 6. Никаких "защитных" try/catch

Не оборачивай работающий код в `try { ... } catch (Exception e) { log; }`. Это
прячет следующий баг и усложняет диагностику.

Исключение: если исходный код уже использует такой паттерн и фикс расширяет его. В
этом случае — следуй стилю файла (принцип 4).

### 7. Не добавляй комментарии "что"

Не пиши:
```java
// Check if email is null  ← плохо, видно из кода
if (email == null) return null;
```

Пиши только если есть нетривиальное "почему":
```java
// Регистр e-mail приводим в нижний — в БД они уникальны без учёта регистра (см. STOR-450).
return email.toLowerCase();
```

И только если *читатель файла* реально не догадается. В большинстве случаев — не
пиши вообще.

## Процесс реализации

### Шаг 0. Что нужно на входе

- Отчёт от `defect-analyzer` или эквивалент: где сломано (`file:line`), root cause,
  edge cases.
- Утверждённый план фикса (одно-два предложения).
- Ключ Jira-задачи (для контекста, не для коммита).

Если чего-то не хватает — попроси:
- Если нет анализа → "вызови defect-analyzer или дай мне file:line и описание".
- Если нет плана → "опиши в одно-два предложения, что хочется сделать".

### Шаг 1. Прочитай файл целиком в зоне правки

Открой метод/класс полностью. Не правь по экстракту из греп — пропустишь стиль и
существующие проверки.

### Шаг 2. Найди минимальное изменение

Если есть несколько вариантов — выбери тот, что меньше всего трогает чужое:
- `null check` в начале метода vs `Optional.ofNullable` обёртка → первое.
- Изменить одну ветку в `switch` vs переписать `switch` → первое.
- Поправить условие в существующем `if` vs добавить новый `if` сверху → то, что
  ближе к стилю файла.

### Шаг 3. Сделай правку через `edit`/`write_file`

`edit` показывает diff автоматически. Пиши целевое состояние, не "до и после".

### Шаг 4. Перепроверь по чек-листу

- [ ] Не изменил публичную сигнатуру?
- [ ] Не ввёл новый класс/интерфейс/метод (без явного согласования)?
- [ ] Не добавил `// TODO` или `// FIXME`?
- [ ] Не добавил `try/catch` "на всякий случай"?
- [ ] Не добавил логирование без причины?
- [ ] Соответствует стилю файла (guard clauses / early return / lombok / builder)?
- [ ] Покрыты edge cases из плана?

Если хоть один пункт нарушен — откати конкретный кусок и переделай. Лучше сделать
больше итераций сейчас, чем разбирать регрессию потом.

### Шаг 5. Если нашёл смежную проблему

НЕ правь. Скажи пользователю одной репликой:
> "По ходу заметил, что в `Foo#bar` похожая проблема с null — оформить отдельной
> задачей?"

И продолжай с изначальным фиксом.

## Что НЕ делать

- Не делай "косметические улучшения" поверх фикса (выравнивание, переименование
  переменных, упрощение условий, не связанные с багом).
- Не пиши тесты — это работа `тестописателя` (отдельный субагент / шаг в
  оркестраторе). Здесь только продакшен-код.
- Не обновляй версии зависимостей.
- Не меняй конфигурационные файлы (`application.yml`, `build.gradle`), если задача
  явно не про них.
- Не добавляй `@Deprecated` на публичные методы.
- Не оборачивай существующие методы в новые методы-обёртки.

---

## Связь с другими скиллами

- `defect-analyzer` — даёт вход (где, почему, edge cases). Без хорошего анализа фикс
  идёт вслепую.
- `java-spring-dev` — справочник по конвенциям Java/Spring (Lombok, layer-first
  packaging, паттерны entity/DTO/service). При необходимости загрузи скилл
  `java-spring-dev` — но не для багфикс-принципов, только за конвенциями.
- `minor-defect-fix` — оркестратор. Вызывает bugfix-developer после одобренного
  пользователем плана.

---

## Автономное использование

Если скилл вызван без оркестратора (просто "почини баг в `Foo#bar`"), задай 3
вопроса одним сообщением и продолжай:

1. Какой файл/метод правим? (если не указан)
2. Какое поведение должно стать?
3. Какие edge cases надо покрыть?

После ответа — иди по процессу.
