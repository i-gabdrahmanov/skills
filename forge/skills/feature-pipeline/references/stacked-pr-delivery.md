# Фаза 6: stacked-PR доставка (по задачам)

Решение проекта — **PR на задачу** + **интеграционная ветка фичи**. Поскольку задачи фичи
обычно зависимы (контроллер не компилится без сервиса, сервис — без сущности), ветки
выстраиваются **stacked**: ветка задачи ответвляется от ветки той задачи, от которой она
зависит. Ревьюер мержит цепочку по порядку — всё стекается в интеграционную ветку фичи,
а из неё одним PR — в default.

## Именование веток

- **Интеграционная ветка фичи**: `feature/<slug>` (slug фичи — обычно ключ Story).
  **Прямые коммиты и push в неё ЗАПРЕЩЕНЫ** — это enforced: gate-guard
  (`branch_protection`) блокирует commit/merge/rebase на ней и любой push в неё.
  Она обновляется ТОЛЬКО мерджем PR сабветок, а создаёт её санкционированный
  `story_branch_push.py` (пушит default-tip на новое имя; коммитов не создаёт,
  существующую ветку не двигает). НЕ пытайся обойти руками.
- Сабветки задач с Jira: `feature/<jira-key>` (ключ ПОДЗАДАЧИ из
  `jira-tasks-result.json`, маппинг `task_id → key`).
- Сабветки задач без Jira: `feature/<slug>-<taskId>` (например `feature/bulk-export-T1`).

## Построение стека

Идёт по топологической сортировке `depends_on`:

```
default (main)
└─ feature/<slug>  (интеграционная: НИКАКИХ прямых коммитов)   target PR → main (мержить последним)
   └─ feature/STOR-201  (T1: entity+repo+migration)            target PR → feature/<slug>
      └─ feature/STOR-202  (T2: dto+mapper+service)            target PR → feature/STOR-201
         └─ feature/STOR-203  (T3: controller)                 target PR → feature/STOR-202
```

Корневые задачи ветвятся от интеграционной ветки (при создании она равна default-tip),
их PR таргетят её. При ветвлении (две задачи зависят от одной) каждая ветвь ответвляется
от общего родителя; их PR независимы между собой, но оба таргетятся на ветку родителя.
Так в default попадает ровно один PR фичи — после мерджа всех задачных PR.

## Идемпотентность: план доставки ПЕРЕД необратимым (обязательно)

Доставка необратима, а Bitbucket-PR **не дедуплицируется**: если прошлый прогон упал после
доставки части задач, повторная доставка вслепую создаст **дубль ветки/PR**. Поэтому ПЕРЕД
гейтами 4–5 всегда считай детерминированный план — что уже сделано:

```bash
python3 <project>/.gigacode/skills/feature-pipeline/scripts/delivery_plan.py \
    "<папка фичи>/task-plan.json" \
    --manifest "<project>/ground/statements/feature-pipeline/<slug>/manifest.json" \
    --root "<project>" --pipeline-config "<project>/ground/pipeline.json" \
    --jira-result "<папка фичи>/jira-tasks-result.json" --json
```

На каждую задачу возвращает `action` (ключ идемпотентности — имя ветки):
- **`skip`** — `07-deliver-<id>` уже completed: задача доставлена, ветку/PR **не трогай**.
- **`resume`** — ветка существует (локально/в origin), но шаг не закрыт: **не пересоздавай
  ветку**, доведи push/PR; перед созданием PR **проверь, нет ли уже открытого** PR для этой
  ветки (Bitbucket MCP — поиск по source-branch), и не создавай второй.
- **`create`** — ничего нет, обычный путь.

Если `summary.all_done=true` — доставлять нечего, переходи сразу к отчёту (гейт 6). Делай
push/PR только для задач `create`/`resume`. `remote_checked=false` (offline) → resume считается
по локальным веткам и manifest; всё равно проверь существующий PR перед созданием.

## Алгоритм

Весь код фичи уже написан и лежит в рабочей копии (фаза 3) и протестирован (фаза 4).
Теперь разносим по веткам и коммитам. **Пропускай задачи со статусом `skip` из плана выше;
для `resume` — не пересоздавай ветку.**

1. Определи default-ветку: `git symbolic-ref refs/remotes/origin/HEAD` или `main`/`master`.
2. Топологически отсортируй задачи по `depends_on`.
3. Для каждой задачи по порядку:
   - создай её сабветку от ветки родителя (для корневых — от default-tip: интеграционная
     ветка при создании указывает туда же);
   - закоммить **только файлы этой задачи** (`task.artifacts` + её тесты + changeset);
     сообщение — в стиле `git log -30`, с ключом Jira, без `Co-Authored-By`.
   - Реализационно проще: сделать ветки последовательно, перенося в каждую только
     относящиеся к задаче изменения (cherry-pick по путям / поэтапный `git add`).
   - **НИКОГДА не коммить на `feature/<slug>`** (интеграционная) — gate-guard заблокирует.

**Гейт 4 (коммиты).** Покажи весь план: задача → сабветка → файлы → сообщение коммита.
Спроси «коммитим?». Только после «да» создавай сабветки и коммить.

**Гейт 5 (push + PR).** Покажи план PR: интеграционная ветка + для каждой задачи
source→target, заголовок, тело (суть + ссылка на Jira-задачу первой строкой).
Спроси «пушим и создаём PR?». После «да»:
- заведи интеграционную ветку фичи (идемпотентно; существующую не двигает):
  ```bash
  python3 <project>/.gigacode/skills/feature-pipeline/scripts/story_branch_push.py \
      --feature <slug> --json
  ```
- push сабветок в порядке зависимостей (родитель раньше потомка);
- через Bitbucket MCP создай PR: target = сабветка-родитель (для корневых —
  `feature/<slug>`);
- последним — PR `feature/<slug>` → default с пометкой «мержить после всех задачных PR»;
- push/PR ветки спеки из фазы 5 (target — default спец-репо).

Запомни URL каждого PR (по `task_id`) — пойдут в отчёт.

## Откат / отмена

Без авто-удаления. Если пользователь прерывает на гейте — `pipeline-state` помнит
созданные ветки/задачи; покажи отчёт «создано: ветки …, Jira …» и оставь решение об
уборке человеку. Не удаляй ветки и не закрывай задачи молча.

Откат пайплайна к более раннему шагу (state + код) — `pipeline-state/scripts/rollback.py`
(R4-гейт; та же политика: сироты Jira/веток/PR только перечисляются). См.
`references/rollback.md`.

## Отчёт (Гейт 6)

Черновик комментария в Story:
```
**Фича:** <title>. <1-2 предложения сути>

**PR по задачам (мержить по порядку):**
1. <jira-key T1> — <url> (→ feature/<slug>)
2. <jira-key T2> — <url> (→ ветка T1)
3. <jira-key T3> — <url> (→ ветка T2)
4. фича целиком — <url PR feature/<slug> → main> (мержить последним)

**Тесты:** покрытие изменённых файлов <NN>% (порог <th>%).
**Спецификация:** <PR/ветка/локально — по статусу спецадаптера>.
```
Покажи целиком, отправь только после «да».

Общие детали Bitbucket/Jira MCP (workspace, repo_slug, создание PR/комментария) —
`../minor-defect-fix/references/{bitbucket-workflow,jira-workflow}.md`.

## Альтернатива (если stacked окажется неудобным)

Открытый вопрос дизайна (README §11): вместо stacked — один прогон доводит **одну**
задачу до PR, остальные отдельными прогонами. Тогда механика проще (все ветки от default,
по очереди после мержа), но «фича» растягивается. Применять, только если пользователь
явно предпочёл этот режим.
