# BRD Grounder

Скилл собирает контекст для бизнес-требований из существующих источников — это замена
веб-поиску на рантайме, где нет инструмента веб-поиска (есть только `web_fetch` по URL).
Скилл **read-only по отношению к исходникам**: он только читает и формирует выжимку.

Цель — чтобы автор BRD (интервьюер или быстрый генератор) опирался на:
- то, что **уже задокументировано** (не дублировал требования),
- **реальную систему** (существующие entity, API, интеграции, ограничения),
- **доменную терминологию** проекта,
а пробелы — добирал у пользователя в интервью, а не выдумывал.

---

## Вход

- Тема/идея будущего BRD (формулировка пользователя или то, что уже прояснилось в интервью).
- Опционально — `<slug>` будущего BRD (для имени файла выжимки).
- Опционально — URL, которые дал пользователь (доки, конкуренты, стандарты).

Из темы выдели 3–6 **ключевых терминов** (сущности, действия, домен) — по ним идёт поиск
во всех источниках.

---

## Источники (собирай по порядку; каждый — с graceful-деградацией)

Для каждого источника: **нет источника → не падай, пропусти и запиши это в раздел
«Пробелы».** Ни один источник не обязателен; ценность — в объединении того, что есть.

### Источник 1. Локальные существующие БТ

Ищи готовые требования в репозитории:

```bash
ls business-requirements/*.md 2>/dev/null
ls docs/feature-pipeline/*/brd.md 2>/dev/null
```

По ключевым терминам найди релевантные файлы (`grep_search` по содержимому). Для каждого
связанного БТ извлеки: путь, заголовок, 1–2 строки сути, пересекающиеся требования и
границы scope. Цель — **не дублировать**: если тема уже покрыта, скажи об этом явно и
предложи дополнить существующий документ, а не писать новый. Проставляй ссылки `[[<slug>]]`
на связанные БТ.

### Источник 2. Confluence (через Atlassian MCP)

Требования часто живут в Confluence. Если подключён Atlassian/Confluence MCP — ищи там.

- **Не угадывай имена тулов.** Проверь список доступных инструментов и найди по шаблонам:
  поиск — `mcp__*confluence*search*` / `mcp__*atlassian*search*` (часто принимает **CQL**,
  напр. `text ~ "<ключевые термины>" AND type = page`), чтение тела —
  `mcp__*confluence*get*page*` / `mcp__*atlassian*get*page*`.
- Возьми 3–5 наиболее релевантных страниц: заголовок, ссылку, краткое резюме, ключевые
  требования/решения. Тело может быть в storage-format/ADF — извлекай смысл, не верстку.
- **Если Confluence MCP не подключён** — не падай. Запиши в «Пробелы»: «Confluence не
  подключён — существующие требования в Confluence не проверены» и, если уместно, спроси
  у пользователя ссылку на пространство/страницу (её можно прочитать через `web_fetch`).

(Стиль работы с MCP — как в `~/.gigacode/skills/minor-defect-fix/SKILL.md`, раздел
«Карта инструментов MCP»: по шаблонам, без угадывания, с деградацией.)

### Источник 3. Веб по URL (опционально, `web_fetch`)

Инструмента веб-**поиска** нет. Но если пользователь дал **конкретные URL** (конкурент,
стандарт, регуляторный документ, дока) — прочитай их через `web_fetch` и возьми ключевые
факты. Сам URL'ы не выдумывай и не «ищи». Если для раздела «Рынок/конкуренты» или
«Регуляторика» данных нет — отметь это как пробел/допущение, а не фабрикуй.

---

## Выход

Сохрани компактную выжимку в `ground/brd-grounding/<slug>.md` (создай папку, если нет; `<slug>` —
kebab-case темы). **НЕ клади рядом с BRD** (`business-requirements/`) — это внутренняя рабочая заметка,
а не документ требований; соседство путало (выглядело как второй BRD). Структура:

```markdown
# Grounding: <тема>

**Дата:** <YYYY-MM-DD>
**Ключевые термины:** ...
**Источники:** локальные БТ / Confluence / код / web — (отметь, что реально было)

## Связанные существующие БТ
| Источник | Где | Суть | Пересечение |
|----------|-----|------|-------------|
| локально | business-requirements/x.md | ... | дублирует разд. 6 |
| Confluence | <ссылка> | ... | смежная тема |
*(если ничего не найдено — «Существующих требований по теме не найдено».)*

## Доменный глоссарий
- **Термин** — значение (как употребляется в проекте/БТ).

## Пробелы (добрать в интервью / у пользователя)
- [ ] ... (например: «Рынок и конкуренты — нет данных без веб-поиска, спросить
      пользователя или дать URL»)
- [ ] ... («Confluence не подключён — требования там не проверены»)
```

---

## Результат (что вернуть вызывающему скиллу)

1. Путь к файлу выжимки.
2. Краткая сводка: сколько связанных БТ найдено (и нет ли дубля темы), проверен ли Confluence.
3. **Список «направлений для углублённого интервью»** — те темы, по которым контекста
   не нашлось (рынок, конкуренты, регуляторика, тех-стандарты и т.п.): их `brd-interview`
   превратит в вопросы пользователю, раз веб-поиска нет.
4. Если найден дубль темы — явно предложи дополнить существующий БТ вместо нового.

---

## Что НЕ делать

- Не писать в исходники проекта — только в `ground/brd-grounding/`.
- Не класть выжимку в `business-requirements/` рядом с BRD (путает; это рабочая заметка, не требования).
- Не угадывать имена MCP-тулов — искать по шаблонам, при отсутствии сервера деградировать.
- Не блокироваться на отсутствии любого источника — собрать что есть, остальное в «Пробелы».
- Не имитировать веб-поиск: без URL от пользователя в интернет не ходим (`web_fetch` — только
  по данным ссылкам).
