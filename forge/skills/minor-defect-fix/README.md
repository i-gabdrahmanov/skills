# minor-defect-fix

Скилл закрывает полный цикл работы с минорной задачей из Jira: разбирает тикет,
находит место правки в коде, чинит, актуализирует тесты до 80% покрытия, обновляет
спецификацию в отдельном репо, готовит отчёт в Jira и аккуратно коммитит — с
возможностью создания двух независимых PR в Bitbucket: для кода и для спеки.

## Для кого

- Разработчик на Java/Spring Boot (Gradle или Maven).
- Подключены MCP-серверы для **Atlassian (Jira)** и **Bitbucket**.
- Репозиторий спецификации лежит в отдельном git-репо в .md-формате.

## Архитектурный принцип

Главный агент держит все необратимые действия (коммиты, push, общение с Jira). Работа
разделена на два слоя: **вложенные скиллы** (загружаются в контекст главного агента —
он действует по их инструкциям) и **субагенты** (изолированный контекст, возвращают
структурированный отчёт).

**Вложенные скиллы:**

| Скилл | Зона ответственности |
|---|---|
| `defect-analyzer` | анализ задачи и поиск места правки в коде/спеке |
| `bugfix-developer` | реализация фикса с принципами багфикса (минимальное изменение, сохранение сигнатур) |
| `java-spring-dev` *(опционально)* | Java/Spring конвенции — подгружается при необходимости |

**Субагенты:**

| Субагент | Тип | Зона ответственности |
|---|---|---|
| тестописатель | `general-purpose` | актуализирует и пишет тесты под фикс |
| тестраннер | `general-purpose` | прогоняет тесты + JaCoCo, возвращает JSON |
| спецадаптер | `general-purpose` | правит .md в спец-репо, создаёт ветку `feature/<JIRA-KEY>` |

**Когда что.** Скилл выбирают для шагов с тесной интеракцией с пользователем — анализ
может задавать вопросы, разработчик — уточнять edge cases. Субагент — для шагов с
тяжёлым выводом (gradle, JaCoCo, скан спеки), который не должен оседать в основном
диалоге. `defect-analyzer` и `bugfix-developer` работают и автономно — можно вызывать
их без оркестратора (`"разбери STOR-1234"` / `"почини этот баг"`).

## Что в скоупе

Минорные баги: `issuetype = Bug`, `priority ∈ {Minor, Low, Trivial}`, одна-две
затронутые области, точечные правки.

## Что НЕ в скоупе

Эпики, истории, рефакторинги, миграции, breaking-changes — скилл срабатывает на
скоуп-чеке и **останавливается до подтверждения**, чтобы не начинать большую правку
«молча».

## Гарантии безопасности

- Никаких `git push --force`, `git reset --hard`, `--no-verify`, `--amend` без явного
  запроса пользователя.
- Каждый необратимый шаг требует явного «да»: комментарий в Jira, коммит, push кода,
  PR кода, push спеки, PR спеки — это шесть отдельных точек.
- Главный агент сам правит код через `edit`/`write_file` — пользователь видит каждый diff,
  субагенты не трогают production-код.

## Установка

Все три скилла лежат рядом в `~/.gigacode/skills/` (user-уровень) и в
`.gigacode/skills/` проекта. Оркестратор знает имена `defect-analyzer` и
`bugfix-developer` и подгружает их по ходу.

```
~/.gigacode/skills/
├── minor-defect-fix/            # оркестратор
│   ├── SKILL.md                 # основной workflow
│   ├── README.md                # этот файл
│   ├── references/
│   │   ├── jira-workflow.md     # детали Jira MCP
│   │   ├── bitbucket-workflow.md # создание PR (код и спека)
│   │   └── coverage.md          # парсинг JaCoCo XML
│   └── scripts/                 # зарезервировано
├── defect-analyzer/             # вложенный скилл анализа
│   └── SKILL.md
└── bugfix-developer/            # вложенный скилл реализации фикса
    └── SKILL.md
```

## Конфиг

При первом запуске скилл спрашивает путь к git-репо со спецификацией и сохраняет в
`~/.gigacode/skills/minor-defect-fix/config.json`:

```json
{
  "projects": {
    "/Users/.../StorageService": {
      "docs_path": "/Users/.../StorageService-spec"
    }
  }
}
```

Маппинг по корню репо кода — для каждого проекта свой путь к спеке.

# Сценарии применения

## 1. Happy path — простой баг с фиксом и обновлением спеки

**Запрос:** «Почини STOR-1234, минорный баг с null email в UserService.»

1. Главный читает конфиг → путь к репо спеки уже сохранён.
2. Через Atlassian MCP подтягивает задачу: `priority=Minor`, `type=Bug` → скоуп-чек
   проходит.
3. Подключает скилл `defect-analyzer`, который возвращает отчёт:
   `UserService.java:42` бросает NPE, тест `UserServiceTest#shouldGetEmail`
   покрывает только happy path, в спеке `domain/users.md` есть раздел про
   `getByEmail`.
4. Главный показывает план → пользователь говорит «да».
5. Подключает скилл `bugfix-developer` и правит код через `edit` по его принципам
   (минимальное изменение, сохранение сигнатуры `getEmail()`).
6. **Тестописатель** обновляет существующий тест и пишет новый на null-кейс.
7. **Тестраннер**: 12 passed, 0 failed, покрытие 88%.
8. **Тестраннер второй раз** — полный pre-commit: build ok, tests ok, spotless ok,
   coverage ok.
9. **Спецадаптер** создаёт ветку `feature/STOR-1234` в репо спеки, правит
   `domain/users.md`, коммитит локально.
10. Главный показывает diff спеки → пользователь говорит «push + PR».
11. Главный пушит спеку, создаёт PR через Bitbucket MCP.
12. Главный показывает сообщение коммита кода в стиле проекта → «да».
13. Коммитит, пушит, создаёт PR кода.
14. Финальный Jira-отчёт со всеми ссылками: ветка/PR/коммит кода + PR спеки →
    «отправить».

**Итог:** два PR в Bitbucket, комментарий в Jira со всеми ссылками. Пользователь
подтвердил 4 точки: план → push спеки → коммит → отправка отчёта.

---

## 2. Первый запуск — настройка конфига

**Запрос:** «Сделай STOR-980, это первый раз скилл запускаю.»

1. Главный читает `config.json` → нет записи для текущего корня.
2. Спрашивает: *«Где лежит git-репо со спецификацией проекта (.md)?»*
3. Пользователь даёт путь.
4. Главный проверяет: папка существует, есть `.git`, есть `.md`.
5. Сохраняет конфиг.
6. Дальше — обычный поток (сценарий 1).

**Итог:** настройка занимает один вопрос, следующие запуски в этом проекте идут уже
без него.

---

## 3. Скоуп-чек — задача оказалась крупнее

**Запрос:** «Возьми STOR-5500.»

1. Главный читает Jira: `priority=Critical`, `type=Story`, в описании «refactor
   authentication to OAuth2 with breaking API changes».
2. Скоуп-чек срабатывает по трём пунктам (priority/type/keywords).
3. Главный пишет:
   > *«Задача выглядит крупнее минорного дефекта (Critical/Story, breaking change в
   > описании). Это правда минор? Продолжаем или останавливаемся?»*
4. Если пользователь говорит «нет» — скилл завершается без правок.
5. Если «продолжай, понимаю риски» — поток идёт, но с явным предупреждением, что план
   и тесты будут шире.

**Итог:** пользователь не «случайно» отдал скиллу эпик в надежде на минорную правку.

---

## 4. Внутренний фикс — спека править не нужна

**Запрос:** «STOR-2100, поправь утечку в кэше провайдеров.»

1. Скилл идёт обычным путём до раздела «Спецификация».
2. **Спецадаптер** читает diff — это внутренняя переменная `cacheRef`, не упоминается
   ни в одном .md.
3. Возвращает `{"no_changes": true, "uncovered_in_spec": []}`.
4. Главный сообщает пользователю одной строкой:
   > *«В спеке нет раздела про эту часть — править нечего.»*
5. Идёт к коммиту кода. Ветка `feature/STOR-2100` в репо спеки не создаётся.
6. В Jira-отчёте блок спеки:
   > *«Правки спеки не требуются (затронутая часть не описана).»*

**Итог:** скилл не плодит мусорные ветки в спец-репо без необходимости.

---

## 5. Тесты падают после первого фикса — итерация субагентов

**Запрос:** «STOR-3300.»

1. Главный делает фикс.
2. **Тестописатель** правит тесты под новое поведение.
3. **Тестраннер**: 8 passed, **2 failed** — `OrderServiceTest#shouldChargeWithDiscount`,
   `PricingTest#shouldApplyCoupon`.
4. Главный понимает: фикс задел расчёт скидок, тестописатель не понял связь.
5. Главный второй раз зовёт **тестописателя** с уточнением: «эти два теста падают,
   разберись — править их или фикс некорректен».
6. Тестописатель: «фикс корректен, тесты ожидали старое поведение — обновил оба».
7. **Тестраннер** второй раз: всё зелёное, покрытие 84%.
8. Дальше как обычно.

**Лимит:** 3 итерации тестописатель↔тестраннер. На третьей всё ещё не зелёное → скилл
останавливается и показывает проблему пользователю.

---

## 6. Покрытие не дотягивает до 80%

**Запрос:** «STOR-450, добавь null-чек в маппер.»

1. Фикс — одна строка, добавляется ветка `if (input == null) return null;`.
2. **Тестописатель** пишет один тест на новую ветку.
3. **Тестраннер**: все тесты зелёные, но покрытие изменённого файла **73%** (старого
   долга много).
4. Главный зовёт **тестописателя** второй раз: «допиши тесты на непокрытые методы, не
   выходи за пределы изменённого файла».
5. Тестописатель пишет ещё 3 теста.
6. **Тестраннер**: покрытие 84% → ок.

Если бы класс был DTO/конфиг и 80% объективно недостижим — главный сообщает
пользователю и предлагает оформить долг по тестам отдельным тикетом, **без**
натягивания искусственных тестов ради цифры.

---

## 7. Отказ от PR — только локальный коммит

**Запрос:** «STOR-880, сделай как обычно, но я хочу пока без push.»

1. Всё до коммита — стандартно.
2. На вопросе про спеку пользователь говорит «оставить локально» → ветка спеки осталась
   только в репо-копии.
3. На вопросе про PR кода — «нет, пока не пушим».
4. Финальный Jira-отчёт:
   - **Код:** ветка `feature/STOR-880`, коммит `a3f8c12`, локально (не запушен).
   - **Спецификация:** локальная ветка `feature/STOR-880` в `<docs_path>`, не
     запушена.
5. Пользователь сам потом запушит обе ветки и создаст PR-ы в удобный момент.

## Точки подтверждения пользователя

В полном happy-path сценарии у пользователя шесть точек, где скилл останавливается и
ждёт «да»:

1. **План фикса** — перед правкой кода.
2. **Diff спеки** — куда дальше: push+PR / только push / локально / откатить.
3. **Сообщение коммита кода** — соответствует стилю проекта, можно поправить.
4. **Создание PR кода** — да/нет.
5. **Черновик Jira-комментария** — отправить / отредактировать / не отправлять.
6. **Скоуп-чек** (если сработал) — продолжать или остановиться.

В сценариях с дозапросом (тесты падают, покрытие низкое, нужны уточнения по
неоднозначной задаче) скилл может задать дополнительные вопросы, но никогда — про уже
утверждённый план.
