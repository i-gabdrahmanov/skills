# SDD — Specification-Driven Development документ (PDLC v3.5)

В v3.5 **спецификация первична, код производен** (стр. 66). SDD-документ устроен так, чтобы агент
получил его и реализовал задачу **без дополнительных вопросов**. Скилл `sdd` генерирует `sdd.md`
из BRD рядом с будущим `tech-design.md` в папке фичи. Tech-design затем проектирует ПО `sdd.md`
и проставляет `sdd_ref` в каждую задачу task-plan.

## Обязательные разделы `sdd.md`

```markdown
# SDD: <фича>

**Jira:** <KEY, напр. KIDPPRB-8639 — ПЕРВОЙ строкой, если задача из Jira; иначе опусти строку>
**BRD:** ./brd.md

## 1. Бизнес-контекст
Зачем фича, какую проблему решает, кто пользователь. (2–5 предложений, из BRD.)

## 2. Функциональные требования (Given-When-Then)
По каждому сценарию — приёмочный критерий в формате:
- **Given** <предусловие> **When** <действие> **Then** <ожидаемый результат>
Покрыть и happy-path, и ошибочные ветки.

## 3. Нефункциональные требования (NFR)
Производительность, нагрузка, безопасность, идемпотентность, лимиты, SLA — что применимо.

## 4. API-контракты
Для каждого нового/изменённого эндпойнта: метод, путь, request/response схема, коды ошибок.
Для Kafka: топик, ключ, схема сообщения, направление (producer/consumer).

## 5. Модель данных
Затрагиваемые entity/таблицы, новые поля — на уровне «что меняется». БЕЗ Liquibase-changeset,
имён классов и пакетов: реализационные детали миграций — забота фазы tech-design.

## 6. Критерии приёмки (Acceptance)
Сводный чек-лист «сделано», по которому собирается evidence bundle. Каждый пункт —
проверяемое утверждение Given-When-Then; на него позже сошлётся задача task-plan через `sdd_ref`.

## 7. Риски и допущения
Что может пойти не так, какие предположения сделаны.
```

## Якоря для трассировки

Размечай разделы так, чтобы tech-design мог сослаться на них из `sdd_ref` задачи
(например `docs/feature-pipeline/<slug>/sdd.md#t1` или якорь на конкретный сценарий §2/§6).
Каждой будущей задаче должен соответствовать ≥1 пункт Given-When-Then из §2/§6 — это даёт
прямую трассировку требование → задача → тест → evidence.

## Зачем строго

Строгий SDD = меньше «додумывания» агентом, выше autonomy-уровень (агент реализует без вопросов),
и прямая трассировка требование → задача → тест → evidence. Это ядро Intent Loop (стр. 59).
Гейт `check_sdd_doc.py` проверяет наличие `sdd.md` с обязательными секциями и хотя бы одним
сценарием Given-When-Then; без этого фаза `02-sdd` не закрывается. Гейт также **ВАЛИТ** при
наличии код-блоков (```` ```java/kotlin/sql/diff/xml ````) или сигнатур кода
(`import …;`, `@RestController`/`@Service`/`@Entity`, `public class/interface/enum`) —
SDD не содержит реализацию; Liquibase-фрагмент даёт лишь предупреждение.
