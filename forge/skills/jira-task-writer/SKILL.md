# Jira Task Writer

> **Все пути — в `feature-pipeline/references/skill-paths.json` (секция `skills.jira-task-writer`).**
> Не используй `~/.gigacode/...` — читай из конфига.

Скилл превращает **план работ** в **задачи Jira**: одну Story на фичу и по одной
Sub-task на каждую задачу из `task-plan.json`. Это шаг постановки задач в пайплайне
разработки фичи (фаза 2.5 в `feature-pipeline`), но работает и автономно.

**Главный принцип: создание задач необратимо, поэтому только после явного «да».**
Сначала собери полный черновик всех задач, покажи его пользователю целиком, и лишь
затем создавай. Никаких «по ходу создам, потом покажу».

**🚨 ЗАПРЕЩЕНО использовать `ask_user_question` в любом режиме.** Ты — генератор
черновика, а не диалоговый агент. Если тебе нужно что-то уточнить у пользователя
(например, какой Epic привязать или добавить ли в спринт) — верни `pending_questions`
в JSON-результате (см. шаг 5). Оркестратор задаст эти вопросы и передаст ответы при
повторном запуске. Никаких исключений — ни для субагента, ни для «автономного» режима.

---

## Вход

1. **`task-plan.json`** — машиночитаемая разбивка (контракт `tech-design`). Из него:
   `title` (заголовок Story), массив `tasks` (каждая → Sub-task), `depends_on` (порядок),
   `acceptance` (в описание Sub-task).
2. **`brd.md`** — для тела Story: контекст (§1), цель (§3), критерии приёмки (§12).
3. **`pipeline.json`** (опционально) — единый конфиг проекта `<project>/ground/pipeline.json`.
   Из секции `jira` берутся настройки проекта: ключ, типы задач, кастомные поля,
   метки, доска. Если файл передан — используй его; если нет — запрашивай у пользователя
   только ключ проекта и валидируй его (`[A-Z]+`).
4. **Конфиг Jira-проекта** (ручной) — ключ проекта (например `STOR`), кому назначать,
   метки. Оркестратор передаёт; при автономном вызове спроси один раз, что неизвестно,
   и валидируй ключ проекта (`[A-Z]+`).

Если `task-plan.json` не передан, а есть только идея/BRD — это не твоя задача: сначала
нужен `tech-design`. Сообщи об этом, не выдумывай разбивку сам.

### Приоритет конфигурации
1. Прямые параметры от оркестратора (assignee, labels — высший приоритет).
2. `pipeline.json` → `jira.*` (средний приоритет).
3. MCP-запрос `createmeta` (fallback, если нет в pipeline.json).
4. Запрос у пользователя (низший приоритет — если нет ни конфига, ни MCP).

Схема полей `jira` в pipeline.json описана в `references/jira-config-schema.md`.

## Рабочий процесс

### 1. Определи схему проекта (из конфига или MCP)

Не угадывай типы задач. Используй конфиг из `pipeline.json` (если доступен), иначе — MCP.

**Порядок действий:**

1.1. Если передан `pipeline.json` — прочитай его, достань секцию `jira`.
     Из неё доступны: `project_key`, `issue_type_story`, `issue_type_subtask`,
     `issue_type_epic`, `issue_type_bug`, кастомные поля (`epic_link_field`,
     `sprint_field`, `system_field`, `br_st_field` и т.д.), `board`.
     Если поля заполнены (`auto_discovered: true`) — используй их напрямую
     (шаг MCP пропускается).

     Если `auto_discovered: false` — в конфиге только `project_key`; остальное
     определи через MCP (см. 1.2).

1.2. Если `pipeline.json` нет, или в нём нет нужных полей — через MCP узнай доступные
     типы issue для проекта (ищи инструмент вида `*issue*types*`, `*project*meta*`).
     Подбери:
     - тип верхнего уровня для фичи (Story / История / Task, если Story нет);
     - тип подзадачи (Sub-task / Подзадача), если в схеме включены подзадачи.

1.3. Если подзадач в схеме нет — сделай Story + обычные Task со ссылкой «relates to»
     на Story (зафиксируй это в черновике, чтобы пользователь знал).

### 2. Собери черновик (с учётом конфига из pipeline.json)

Если доступен `pipeline.json` → `jira.*` с `auto_discovered: true`, используй
обнаруженные поля:
- **system_field** (cf-идентификатор) — если задан, предложи заполнить
  (например названием подсистемы).
- **epic_link_field** — если задан, спроси: «Привязать к Epic? Укажи ключ Epic».
- **sprint_field** — если задан, спроси: «Добавить в спринт?».
- **board.name** — для подсказки пользователю о названии доски.

Если `auto_discovered: false` — используй только `project_key`, остальное
определяй через MCP (см. шаг 1.2).

**Конвенции проекта («единый ключ»).** Если в конфиге есть `jira.conventions` (выведены из
недавних issue проекта), **используй их как ДЕФОЛТЫ** черновика, чтобы задачи были в едином стиле:
`common_components` → компоненты, `common_labels` → лейблы, `frequent_epic` → предлагаемый Epic Link,
`summary_prefix` → префикс заголовка (напр. `[KID]`). Это дефолты, а не жёсткие правила — пользователь
в черновике может переопределить.

**Story:**
- `summary` = `task-plan.title` (с префиксом из `conventions.summary_prefix`, если он есть в проекте).
- `description` = краткий контекст из BRD §1, цель §3, и общий список критериев приёмки
  §12. Ссылку на BRD/tech-design добавь, если они доступны по URL; иначе укажи путь.
- `components` — из `conventions.common_components` (дефолт), если проект их использует.
- `labels` — `conventions.common_labels` (дефолт) + прямые параметры оркестратора; или уточни.
- `Epic Link` — **ВСЕГДА** включай вопрос про Epic в `pending_questions` при создании Story
  (пользователь может ответить «нет»). Если есть `conventions.frequent_epic` — **предложи его как
  дефолт** в вопросе. Не пропускай вопрос, даже если `epic_link_field` не авто-обнаружен — ключ поля
  возьмёшь из `epic_link_field` или через MCP при создании.
  Пропускай вопрос только если ответ на `epic` уже пришёл в `answers`.
- `Sprint` — если `sprint_field` определён, спроси про спринт.

**Sub-task на каждую задачу** (в порядке топологической сортировки по `depends_on`):
- `summary` = `task.title`.
- `description` = что делаем + список `acceptance` чек-боксами + затронутые слои/артефакты.
- порядок и зависимости: если в схеме есть связи задач — отрази `depends_on` как
  «is blocked by»; если нет — просто упорядочь и упомяни зависимость в описании.
- `components`/`labels` — наследуй от Story (по `conventions`), чтобы вся фича была в едином ключе.

### 3. Черновик — единственный выход

Ты НЕ спрашиваешь пользователя. Ты собираешь черновик (см. шаг 2) и возвращаешь его
в JSON по контракту (см. шаг 5). Оркестратор сам покажет черновик пользователю
и получит подтверждение. Твоя задача — только собрать и вернуть данные.

Если оркестратор перезапустил тебя с полем `revision` (правка от пользователя) —
пересобери черновик с учётом правки (измени число/состав подзадач, объедини, переименуй,
убери) и снова верни `created: false` с обновлённым `draft`. Этот черновик — новый источник
истины для последующего создания; не возвращайся к разбивке из task-plan.

### 4. Создание (с кастомными полями из конфига)
Оркестратор передаёт флаг `confirmed: true` и, опционально, `answers` на `pending_questions`.
Только в этом случае — создавай задачи.

**🔁 Идемпотентность (искать перед созданием).** Создание Jira-задачи необратимо, а перезапуск
после частичного падения легко задвоит задачи. Прежде чем создавать:
1. Если `jira-tasks-result.json` уже есть — это ledger: для задач с `key` ничего не создавай,
   дозаводи только недостающие (см. ниже).
2. На КАЖДОЙ создаваемой задаче ставь маркер-метки **`forge:<feature-slug>`** (на Story и все
   Sub-task) и **`forge-task:<task_id>`** (на соответствующий Sub-task). Это стабильный ключ
   идемпотентности, не зависящий от ledger.
3. Перед созданием Story/Sub-task **сначала ищи по метке** (JQL `labels = "forge:<slug>"` и
   `labels = "forge-task:<task_id>"`). Нашёл — **переиспользуй найденный key**, не создавай
   второй. Так перезапуск безопасен, даже если `jira-tasks-result.json` потерян до записи.

Записывай ledger `jira-tasks-result.json` **инкрементально** — сразу после каждого созданного
issue, а не только в конце, чтобы падение на середине не стёрло уже сделанное.

**🚨 Создавай РОВНО подтверждённый черновик, а не сырой `task-plan.json`.** Если на прошлом
шаге пришёл `revision` (например «нужна 1 задача вместо 4») — черновик уже пересобран, и
число/состав Sub-task берётся из него. Не воссоздавай по одной подзадаче на каждую задачу
task-plan, если пользователь сократил/объединил их. По порядку:

1. Создай Story (тип из шага 1, проект — из `project_key` конфига или от оркестратора,
   summary, description, метки/assignee если заданы).
   **Если в конфиге есть кастомные поля** (через `epic_link_field`, `sprint_field`,
   `system_field` и т.д.) — передай их при создании:
   - `epic_link_field` → значение Epic Link (если пользователь указал).
   - `sprint_field` → ID активного спринта (если пользователь указал).
   - `system_field` → значение, которое указал пользователь.
   - `br_st_field`, `fr_st_field` → ссылки на спецификацию, если есть.
   Запомни её `key`.
2. Для каждой задачи создай Sub-task с `parent = story.key` (для подзадач родитель
   обязателен). Если тип — обычная Task, проставь связь со Story вместо parent.
   При создании Sub-task тоже используй кастомные поля из конфига (особенно `system_field`).
3. Проставь связи «blocked by» по `depends_on`, если схема поддерживает.

**При ошибке на N-й задаче — остановись и верни, что уже создано.** Не откатывай
созданное молча и не продолжай вслепую. Пользователь решит: дозавести вручную или
повторить.

### 5. Верни результат

JSON по контракту (см. `contracts.md §3` в feature-pipeline):

**Для не подтверждённого черновика (всегда при первом запуске):**
```json
{
  "created": false,
  "draft": {
    "story": {
      "summary": "Автозакрытие пустых регулярных задач с employee",
      "description": "## Контекст\n...\n### Критерии приёмки\n- ..."
    },
    "subtasks": [
      {
        "task_id": "T1",
        "summary": "Repository: выборка задач с employee + проверка пустоты",
        "description": "**Что делаем**\n...\n**Acceptance criteria**\n- [ ] ...",
        "depends_on": []
      }
    ]
  },
  "pending_questions": [
    {
      "id": "epic",
      "question": "К какому Epic привязать Story? Укажи клюк Epic (например EPIC-123) или 'нет'."
    },
    {
      "id": "sprint",
      "question": "Добавить в спринт? Укажи ID спринта или 'нет'."
    }
  ],
  "skipped": false
}
```

Если `pending_questions` не пуст — оркестратор задаст вопросы, получит ответы
и перезапустит субагента с полем `answers: {"epic": "...", "sprint": "..."}`.
Субагент должен обработать ответы и вернуть обновлённый `draft` (уже без `pending_questions`,
если всё ясно, или с новыми вопросами, если нужно уточнить ещё).

**Для подтверждённого запуска (`confirmed: true`) — после создания:**
```json
{
  "created": true,
  "story": {"key": "STOR-200", "url": "https://.../STOR-200"},
  "subtasks": [
    {"task_id": "T1", "key": "STOR-201", "url": "..."},
    {"task_id": "T2", "key": "STOR-202", "url": "..."}
  ],
  "skipped": false
}
```

Маппинг `task_id → key` критичен: по нему фаза доставки привяжет ветки/PR к задачам.
Если создание пропущено или MCP недоступен — `created: false`, `skipped: true`, и поясни
пользователю причину.

Сохрани этот JSON в `<папка фичи>/jira-tasks-result.json`. Затем — **самопроверка
паритета (gate):**
```bash
python3 <project>/.gigacode/skills/jira-task-writer/scripts/check_jira.py \
    "<папка фичи>/task-plan.json" --result "<папка фичи>/jira-tasks-result.json" \
    --pipeline-config "<project>/ground/pipeline.json"
```
Проверяет: создано 1 Story + по Sub-task на каждый `task_id` (при `skipped`/jira off — skip).
При `fail` — назови задачи без ключа и дозаведи недостающее.

## Если MCP Jira не подключён

Не падай и не выдумывай ключи задач. Сообщи: «MCP Atlassian не настроен — задачи в Jira
не созданы». Верни `skipped: true`. В контексте `feature-pipeline` это переводит прогон
в режим «без Jira»: план остаётся в `task-plan.json`, доставка пойдёт без привязки к
ключам.

Карта инструментов MCP и различия схем — в
[`references/jira-create-workflow.md`](references/jira-create-workflow.md).

## Что НЕ делать

- **НИКОГДА не использовать `ask_user_question`.** Даже если кажется, что «ну вот сейчас точно надо» — нет, не надо. Верни `pending_questions` в JSON.
- **Не создавать ничего без флага `confirmed: true`.** Это внешняя необратимая операция.
- **Не угадывать имена типов** Story/Sub-task — спроси схему через MCP.
- **Не комментировать и не закрывать задачи** — это фаза 6 оркестратора (финальный отчёт).
- **Не придумывать разбивку**, если нет `task-plan.json` — это работа `tech-design`.
- **Не откатывать частично созданное молча** при ошибке — верни факт и отдай решение
  пользователю.
- **Не дублировать задачи** при повторном запуске: если есть флаг `confirmed: true` и
  `jira-tasks-result.json` уже существует — дозаведи недостающее, не создавай заново.
