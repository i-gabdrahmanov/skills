# Defect Analyzer

Скилл разбирает минорный дефект и выдаёт структурированный отчёт: где сломано
(`file:line`), почему (root cause), что затрагивает в тестах и спеке, какие edge
cases надо учесть, что ещё неясно — задать пользователю.

**Скилл не правит код.** Он только читает, грепает и формирует отчёт. Правкой
занимается `bugfix-developer`.

---

## Входные данные

Если скилл вызван из оркестратора `minor-defect-fix` — все входы уже у главного
агента:

- объект Jira (summary, description, comments, priority, type),
- корень репо кода,
- путь к репо спеки (`docs_path`).

Если вызван автономно (`"разбери STOR-1234"`) — собери сам:

1. Через Atlassian MCP подтяни задачу (ищи доступные инструменты
   `mcp__*jira*get*issue*` / `mcp__*atlassian*issue*`). Если MCP не подключён —
   попроси пользователя вставить текст задачи.
2. Корень репо кода: `git rev-parse --show-toplevel`.
3. Путь к спеке: посмотри в `<project>/.gigacode/skills/minor-defect-fix/config.json` (там
   маппинг `{ "projects": { "<repo>": { "docs_path": "..." } } }`). Если нет — спроси
   у пользователя или работай без спеки.

---

## Процесс

### 1. Сформулируй "что → ожидается"
Прочти description и последние 3-5 комментариев. Одно предложение:
> "Метод X в файле Y бросает NPE на null email, ожидается возврат Optional.empty()."

### 2. Найди место правки
- Греп по именам классов/методов из описания и комментариев.
- Греп по тексту ошибок/стектрейсов из задачи и attachments.
- Если упомянут endpoint — найди контроллер по `@RequestMapping`/`@GetMapping`.
- Если упомянуто доменное понятие — найди соответствующий `.md` в спеке по заголовку.

Иди прицельно. Не читай весь репо. Если class A вызывает class B и B не имеет
отношения к багу — не читай B.

### 3. Root cause, не симптом
Сформулируй почему конкретная строка приводит к описанной ошибке. Помечай уровень
уверенности явно:
- `[подтверждено]` — root cause очевиден из кода.
- `[гипотеза]` — есть несколько вариантов, нужен запуск/уточнение.

Не выдавай гипотезу за факт. Это потом ловит `bugfix-developer` и пишет неправильный
фикс.

### 4. Затронутые тесты
Найди существующие тесты, покрывающие проблемный метод:
- `grep -r 'ClassName' --include='*Test.java'`
- по именам методов

Не читай тестовые файлы целиком. Достаточно имени теста и краткого given/then.

### 5. Затронутые разделы спеки
Прогрепай `.md` в `docs_path` по именам классов/методов/доменных понятий. Перечисли
конкретные файлы (`api/users.md`, `domain/auth.md`). Если упоминаний нет — так и
зафиксируй: "в спеке этой части нет".

### 6. Edge cases
Что ещё может сломаться от наивной правки? Минимум:
- null / пустые значения,
- граничные значения (0, max),
- одновременный доступ (если код многопоточный),
- сценарий из самой задачи + 1-2 родственных.

### 7. Открытые вопросы
Если что-то в задаче неясно — собери ВСЕ вопросы в один нумерованный список. Не
задавай по одному.

## Формат отчёта

```
## Анализ STOR-XXXX

**Что сломано → ожидается:** одно предложение.

**Где:**
- `src/main/java/.../Foo.java:42` — конкретный метод/строка
- `src/main/java/.../Bar.java:88` — если правка в нескольких местах

**Root cause:** `[подтверждено | гипотеза]`
Краткое объяснение — почему именно тут.

**Затронутые тесты:**
- `FooTest#shouldDoX`
- `BarTest#shouldHandleY`
- (или "тестов нет — это важный сигнал")

**Затронутые разделы спеки:**
- `domain/foo.md#user-flow`
- (или "нет упоминаний в спеке")

**Edge cases для проверки:**
- null email
- пустой список items
- email с UTF-8 эмодзи

**Открытые вопросы:** нет
```

Если есть открытые вопросы:

```
Прежде чем продолжить, уточни:
1. Должен ли метод возвращать Optional.empty() или бросать AccountNotFoundException?
2. Поведение для emails в верхнем регистре — нормализовать или сохранять?
```

И останови работу до ответа. После ответа — выдай отчёт.

---

## Что НЕ делать

- **Не редактируй файлы.** Это скилл анализа.
- **Не предлагай конкретный код фикса.** Это работа `bugfix-developer`. Можешь
  обозначить подход ("guard clause в начале метода"), но не код.
- **Не читай всю документацию подряд.** Иди прицельно по релевантным разделам.
- **Не задавай вопросы по одному.** Собери все в один список — пользователю проще
  ответить разом.
- **Не предполагай, что у тебя есть права на запись.** Скилл может быть запущен в
  read-only режиме (например, на review задачи перед бэклогом).

---

## Связь с другими скиллами

- `minor-defect-fix` — оркестратор, который вызывает defect-analyzer первым шагом
  после скоуп-чека.
- `bugfix-developer` — получает отчёт defect-analyzer как вход и реализует фикс.
  Отчёт должен быть достаточно конкретным, чтобы разработчик не делал вторую
  итерацию анализа.
