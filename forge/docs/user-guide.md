

> **Кому:** инженер, который запускает конвейер на своём проекте.
> **О чём:** что нужно до старта, как запустить, что произойдёт по шагам, где ваши решения,
> как продолжить после обрыва. Обзор для бизнеса — [pipeline-business.md](pipeline-business.md);
> устройство фаз — [pipeline-technical.md](pipeline-technical.md); если что-то пошло не так —
> [troubleshooting.md](troubleshooting.md).

## 1. Что делает конвейер

Проводит **одну новую фичу** от идеи до pull request: BRD → grounding → SDD (спецификация) →
тех-дизайн → задачи в Jira → код (TDD) → покрытие → документация → stacked-PR → отчёт. Между гейтами он
автономен, но всё необратимое (Jira, коммит, push, PR, отчёт) — только после вашего «да».

> Для **минорного бага** есть отдельный, более лёгкий маршрут (`minor-defect-fix`). Этот гайд — про фичу.

---

## 2. Предусловия

- **Java/Spring проект** (Gradle или Maven), вы в корне репозитория кода.
- **Харнес задеплоен в проект:** `<project>/.gigacode/` с `hooks/` и `skills/` рядом (co-located),
  закоммичен в репо.
- **Рантайм запущен с флагом** `--experimental-hooks` — иначе control-plane молчит (см. §4).
- **MCP Atlassian (Jira)** и **Bitbucket** — нужны для фаз 2.5 и 6. Без них пайплайн всё равно идёт
  в режиме «без Jira / до коммита».
- Для доставки до PR проект должен быть **git-репозиторием** (иначе ветки/stacked-PR и ключ
  pipeline-state не работают — конвейер предложит `git init`).

---

## 3. Установка / деплой харнеса

Модель — **проектная**: `hooks/` и `skills/` лежат **co-located** внутри `<project>/.gigacode/`
и закоммичены в репо. Источник правды — каталог `forge/` (его `hooks/` + `skills/`).

Деплой = положить/обновить `.gigacode/` в проекте. Используйте установщик `deploy.sh` —
он копирует hooks+skills (co-located), кладёт `deploy-local.sh` и доводит `settings.json`
(merge блока `hooks` + бэкап) одной командой:

```bash
# из склонированного репо Forge; целевая папка проекта обязательна
cd <forge>
bash deploy.sh /path/to/target-project
```

Резолверы кода выводят базу из фактического расположения файла, так что жёсткой привязки к
домашнему каталогу нет — важна именно **co-location** hooks+skills внутри `.gigacode/`.

📖 Полное руководство (повторный деплой, бэкапы, починка путей после переезда, типовые
сценарии) — [docs/deployment.md](deployment.md).

> ⚠️ Главная ошибка деплоя: положить только скиллы и не влить блок `hooks` в `settings.json` →
> `[HOOK_REGISTRY] 0 hook entries`, весь control-plane молчит. Проверяйте результат через
> `preflight.py` (§4) — он ловит этот класс ошибок.

## 4. Запуск

**Шаг 1 — pre-flight (обязательно).** Убедитесь, что enforcement реально включён:

```bash
python3 <project>/.gigacode/hooks/preflight.py --project .
```
- `exit 0` — харнес активен, можно работать.
- `exit 2` — конфиг не инициализирован (`init_needed`: `ground/pipeline.json` нет/неполон). Нормально
  сразу после деплоя — файл появится при первом запуске пайплайна. Инициализируйте конфиг и прогоните
  `preflight` снова до `exit 0`. Это **не** ENFORCEMENT OFF.
- `exit 1` — **ENFORCEMENT OFF**: хуки не сработают (нет блока `hooks` / `disableAllHooks` / не задеплоено).
  Не ведите пайплайн вслепую — перепроверьте деплой (§3) и запустите `preflight` снова.

> `preflight.py` дополнительно прогоняет доктор целостности пайплайна
> (`skills/feature-pipeline/scripts/doctor.py`) — рассогласование судей, дрейф фаз, битые пути реестра
> (advisory: предупреждение, не блок). Запустить его отдельно: `doctor.py --project . --json`.

**Шаг 2 — запуск рантайма с флагом хуков (всегда):**

```bash
gigacode --experimental-hooks                       # интерактивно
gigacode --experimental-hooks -p "сделай фичу X"     # одной командой
```

Флаг — это аргумент **запуска бинаря**, его нельзя прописать в `settings.json`. Без него —
`[HOOK_REGISTRY] 0 hook entries`.

**Шаг 3 — запустить пайплайн.** Триггер по описанию или явно:

```
сделай фичу <описание>
реализуй фичу end-to-end по STOR-123
запусти feature pipeline
```

## 5. Что произойдёт по шагам и где ваши решения

| Фаза | Что делает конвейер | Ваше действие |
|---|---|---|
| **0 BRD** | Короткое интервью (3–7 вопросов), затем оформляет требования | **Гейт 1:** утвердить BRD; **выбрать критичность** (low/medium/high) |
| **1 Grounding** | Переиспользует или собирает карту системы | Подтвердить переход к спецификации |
| **2 SDD** | Строгая спецификация поведения (`sdd.md`): сценарии GWT, контракты, данные, приёмка | **Гейт SDD:** утвердить спецификацию / запросить правки |
| **2 Design** | Тех-дизайн (вход — `sdd.md`) + `task-plan.json` | **Гейт 2:** утвердить дизайн / запросить правки |
| **2.5 Jira** | Готовит черновик Story + подзадач | Ответить на вопросы (Epic, спринт); **Гейт 3:** «создавать?» |
| **3 Build** | Per task: тесты вперёд (RED) → код (GREEN) | — (автономно, под судьями) |
| **4 Verify** | Догон покрытия + полный прогон | — |
| **5 Document** | Обновляет спецификацию | — |
| **6 Deliver** | Ветки, коммиты, stacked-PR, отчёт | **Гейты 4–6:** коммит / push+PR / отчёт в Jira |

**О критичности (Гейт после BRD).** Определяет, насколько строго форсятся гейты:

| Критичность | Когда | Поведение |
|---|---|---|
| **Низкая** | эксперимент, non-prod, внутр. инструмент | фичекод авто; гейтятся доставка и рискованные пути |
| **Средняя** (дефолт) | обычная прод-фича | commit/push/jira/секьюрные пути — под гейтами |
| **Высокая** | auth / платежи / PII / инфра | почти всё требует подтверждения и доказательств |

> До выбора критичности `gate-guard` блокирует любое рискованное (R2+) действие — это сделано
> намеренно, чтобы шаг нельзя было пропустить.

## 6. Где лежат артефакты

```
<project>/
├── docs/feature-pipeline/<slug>/      # brd.md, tech-design.md, task-plan.json, sdd.md, eval-plan.json
├── docs/system-analysis/              # карта системы (grounding) + grounding-excerpt.json
└── ground/
    ├── pipeline.json                  # конфиг конвейера (критичность, пороги, Jira-мета)
    ├── statements/feature-pipeline/<slug>/   # состояние прогона (манифест, судьи, errors.json)
    └── ai-logs/run-<session>/         # живой лог прогона: agents.log + agents.jsonl (один на прогон)
```

`<slug>` — kebab-case по сути фичи (или Jira-ключ). Спецификация правится в **отдельном** repo
(`docs_path`), не в репо кода.

## 7. Продолжение после обрыва (resume)

Прогресс сохраняется по шагам — после обрыва стрима/процесса продолжайте с места, а не с нуля.

```bash
# какие фичи в работе:
python3 <project>/.gigacode/skills/pipeline-state/scripts/read.py --skill feature-pipeline --list

# состояние конкретной фичи:
python3 <project>/.gigacode/skills/pipeline-state/scripts/read.py --skill feature-pipeline --feature <slug>
```

Снова запустите пайплайн и скажите «резюмируй фичу `<slug>`» — конвейер подхватит незакрытые шаги.
Несколько фич сосуществуют (state намеспейсится по фиче) и не вытесняют друг друга.

## 8. Если судья вернул FAIL

Это нормальная часть процесса. Конвейер сам:
1. Сохраняет проблемы в `judges/errors.json`.
2. **До 3 раз** перезапускает субагента той же фазы с накопленными ошибками — он их исправляет.
3. После 3 неудач — **останавливается и спрашивает вас**: сбросить и начать заново / отменить шаг /
   пропустить гейт вручную (override — только если причина внешняя: нет тестовой БД, недоступен сервис).

Override не подделывает вердикт: FAIL остаётся в логах, снимается лишь блок закрытия, с обязательной
причиной в аудит. Подробные симптомы и команды — в [troubleshooting.md](troubleshooting.md).

## 9. Правила, о которые легко споткнуться

- **Один прогон — одна фича.** Если задача тянет на эпик — конвейер остановится и предложит разбить.
- **Не обходите гейты силой.** Никаких `git push --force` / `git reset --hard` / ручной правки манифеста.
- **Документация — в spec-repo**, код — в репо кода. Не смешивайте.
- **Конвейер не заменяет ревью** — он готовит качественный PR к ревью человеком.
- Если `agent` (субагенты) недоступен в рантайме — конвейер выполнит фазу inline как **деградацию**
  и явно это пометит; это хуже по устойчивости, но не блокер.
