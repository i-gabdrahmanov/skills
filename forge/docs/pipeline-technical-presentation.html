Feature Pipeline

Что именно вызывается на каждой фазе: субагенты, судьи, execution-gates и хуки control-plane.

pipeline > model hooks = enforcement skills = guidance PDLC v3.5
Принцип

Правила форсит рантайм, а не «добрая воля» модели

SKILL.md — guidance

Инструкция для модели. Её можно проигнорировать → одной её недостаточно.

Хуки — enforcement

Рантайм форсит risk ladder, TDD, evidence, token budget, phase-gate. Блок = exit 2 + причина в stderr.

Судьи + gates

Детерминированные проверки (run_judge.py, check_*.py): шаг закрывается только при PASS.

⚠ Запуск обязателен с --experimental-hooks — иначе [HOOK_REGISTRY] 0 hook entries, control-plane молчит. До прогона — preflight.py (exit 0 = активен).

Хук tdd-guard блокирует запись в src/main, пока 04-test-<id> ещё pending. Хук eval-guard блокирует код, пока eval'ы задачи не прошли.

Судьи

9 ступеней проверки качества

brd-judge

БТ языком бизнеса, без кода. LLM + код-токены.

design-judge

Тех-дизайн консистентен с BRD.

eval-judge

Eval'ы покрывают AC, пороги адекватны.

red-judge

Тесты реально падают (RED), не «зелёные стабы».

build-judge

Код зеленит тесты (pass-through, verdict субагента).

reuse-judge

Нет «велосипедов» поверх библиотек/util. LLM + regex по diff.

coverage-judge

JaCoCo ≥ порога (check_coverage.py).

spec-judge

Документация отражает изменения.

delivery-judge

Нет stubs/секретов, Jira консистентна, git чист.

Порядок · Bash

destructive → cost-breaker → evidence → gate-guard → log-agent

Порядок · Write/Edit

pii-boundary → tdd-guard → gate-guard → log-agent

fail-OPEN

Гейт-хук >60с убивается → действие проходит. Поэтому тяжёлые гейты гоняет оркестратор, хуки — лёгкая страховка.

Отказоустойчивость

FAIL → ре-итерация → (override)

PASS → errors.json удаляется, шаг закрывается ↑ judge verdict ↓ FAIL run_judge сохраняет blocking_issues → errors.json (perpetual) ↓ iterations < 3 → перезапуск субагента той же фазы с accumulated_errors ↓ 3 неудачи стоп → спросить человека: сброс / отмена / override (§0.6.1)

Override не подделывает вердикт — FAIL остаётся, снимается лишь блок закрытия, с --reason в аудит. Никогда git push --force / reset --hard / правка манифеста руками.
