# Feature Pipeline — траблшутинг

> **Кому:** инженер, у которого что-то пошло не так во время прогона.
> **Как пользоваться:** найдите симптом → причина → команда/действие. Старт и базовые понятия —
> в [user-guide.md](user-guide.md); устройство фаз — в [pipeline-technical.md](pipeline-technical.md).

## Первая помощь (90% проблем)

```bash
# 1. Харнес вообще активен? (заодно прогоняет доктор целостности пайплайна)
python3 <project>/.gigacode/hooks/preflight.py --project .   # 0=ок, 2=не инициализирован, 1=ENFORCEMENT OFF

# 2. Доктор целостности пайплайна (судьи, фазы, пути реестра)
python3 <project>/.gigacode/skills/feature-pipeline/scripts/doctor.py --project . --json

# 3. Все пути реестра существуют?
python3 <project>/.gigacode/skills/feature-pipeline/scripts/check_paths.py --project .

# 4. Что в работе / на каком шаге застряли?
python3 <project>/.gigacode/skills/pipeline-state/scripts/read.py --skill feature-pipeline --list
```

> Глубже: eval-набор хуков — `bash <project>/.gigacode/hooks/run-hook-tests.sh`; тесты скиллов
> И юнит-тесты хуков (единый прогон) — `python3 <project>/.gigacode/skills/run_all_tests.py`
> (только хуки — `--skill hooks`); метрики из аудита — `hooks/agentops.py`.

## A. Харнес / хуки молчат

### A1. `[HOOK_REGISTRY] 0 hook entries` при старте
**Симптом:** в выводе рантайма строка `0 hook entries`; гейты, TDD, risk-ladder, evidence — не срабатывают.
**Причины и фиксы:**
- **Не передан флаг запуска** → запускайте `gigacode --experimental-hooks` (это аргумент процесса,
  в `settings.json` его прописать нельзя).
- **Блок `hooks` не влит в `settings.json`** (залили только скиллы) → влейте блок из эталона
  `forge/hooks/settings.hooks.json` (см. user-guide §3).
- **`disableAllHooks: true`** в settings → снимите.

### A2. `preflight.py` → exit 1 (ENFORCEMENT OFF)
**Действие:** не ведите пайплайн вслепую. Перепроверьте деплой (user-guide §3) → `preflight.py` снова.
Продолжайте только после exit 0.
**Не путайте с exit 2** (`init_needed`: `ground/pipeline.json` нет/неполон) — это не поломка
энфорсмента, а неинициализированный конфиг. Инициализируйте его и перезапустите preflight до exit 0.

### A3. Гейты не находят скиллы (`../skills`)
**Симптом:** хук/гейт падает, не может найти скилл-скрипт.
**Причина:** нарушена co-location — `skills/` не лежат рядом с `hooks/` в одном `.gigacode/`.
**Фикс:** положите `hooks/` И `skills/` в один дом `<project>/.gigacode/` (см. user-guide §3).
Проверка путей реестра — `check_paths.py --project .`.

### A4. Рантайм «молча скипает» скилл
**Причина:** битый frontmatter (нет `name`/`description`) → скилл считается мёртвым.
**Фикс:** проверьте шапку скилла (`name`/`description` в YAML-frontmatter `SKILL.md`); `doctor.py
--project . --json` подсветит рассогласования реестра/судей.

## B. Гейт-хук блокирует действие (exit 2)

Блокировка — это **штатная защита**, не баг. Читайте `stderr` — там причина.

### B1. `gate-guard`: «критичность не задана» / блок R2+
**Причина:** не пройден гейт критичности после BRD → `gate-guard` блокирует любое R2+ действие.
**Фикс:** ответьте на вопрос о критичности; конвейер запишет `autonomy.{criticality,auto_max_risk}`
в `ground/pipeline.json`. После этого порог применяется per-feature.

### B2. `tdd-guard`: блок записи в `src/main`
**Причина:** шаг `04-test-<id>` ещё `pending` — нельзя писать код, пока RED-тесты не написаны/не падают.
**Фикс:** сначала фаза RED (тесты), `check_tests_red.py` (compile OK + fail), затем код.
**Вариант:** блок `@DataJpaTest`/`@SpringBootTest` при `quality.test_layer=service-unit` — пишите
service-unit + моки, либо переключите на `test_layer=mixed` (escape-hatch).

### B3. `destructive-blocker`: команда в чёрном списке
**Причина:** `rm -rf /`, `git push --force`, `git reset --hard`, `DROP ...` и т.п.
**Фикс:** не обходите проблему деструктивом — разбирайтесь с причиной. Чёрный список не отключается под прогон.

### B4. `pii-boundary`: блок записи
**Причина:** попытка записать PII/секрет вне разрешённого scope.
**Фикс:** уберите секрет из артефакта; секреты не хранятся в pipeline-state и не пишутся в docs.

### B5. `evidence-enforcer`: блок доставки
**Причина:** evidence bundle неполный (completeness ниже `evidence.threshold`) — доставка преждевременна.
**Фикс:** доведите предыдущие фазы (тесты/покрытие/спека) до закрытых шагов, затем доставка.

### B6. `budget-meter`: учёт токен-бюджета
**Симптом:** его нет — хук никогда не блокирует и не предупреждает (circuit-breaker удалён).
**Что делает:** только считает расход по фазам в `<run-dir>/budget.json` и печатает сводку на Stop.
Ориентир бюджета — `quality.token_budget` в `pipeline.json` (справочно, без стопов). Если прогон
дорогой — сжимайте контекст (выжимки шагов вместо полных выводов), тяжёлый вывод — в субагентах.

## C. Судьи и ре-итерация

### C1. Судья вернул FAIL — что делать
**Это штатно.** `run_judge.py` уже сохранил `blocking_issues` в `judges/errors.json` (perpetual).
Конвейер перезапускает субагента той же фазы с накопленными ошибками — **до 3 итераций**.
Сами артефакты руками **не правьте** (это ломает изоляцию и TDD-цикл).

### C2. Зациклился на одном судье (3 неудачи)
**Действие:** конвейер останавливается и спрашивает. Варианты:
- **сбросить** `errors.json` и начать шаг заново;
- **отменить** шаг;
- **override** (§C3) — только если причина FAIL **внешняя и неустранима правкой** (нет тестовой БД,
  внешний сервис недоступен, acceptance ослаблен по согласованию).

### C3. Ручной override гейта (последнее средство)
```bash
python3 <project>/.gigacode/skills/pipeline-state/scripts/override_judge.py \
    --judge <judge-name> --feature <slug> --step-id <step-id> --reason "<почему допустим пропуск>"
# затем закрыть шаг как обычно — update.py увидит override
python3 <project>/.gigacode/skills/pipeline-state/scripts/override_judge.py --feature <slug> --list   # просмотр
```
Override **не подделывает вердикт**: FAIL остаётся в `judges/<judge>.json`, снимается лишь блок закрытия,
`--reason` идёт в аудит. Если на шаге два судьи — override нужен на **каждый** упавший.

### C4. `red-judge` FAIL: «тесты не падают»
**Причина:** RED-тесты зелёные (метод уже частично реализован) → это GREEN, не RED.
**Фикс:** тесты должны падать на **runtime-assert** (`assertThrows`, неверное значение, edge cases),
а не на компиляции. Моки должны верифицировать НЕверное поведение, исправляемое в GREEN.

### C5. `brd-judge` FAIL: «код в требованиях»
**Причина:** BRD написан как спецификация (классы/сущности/SQL).
**Фикс:** перепишите требования языком бизнеса; код-детали идут в `tech-design`, не в BRD.

### C6. `reuse-judge` FAIL: «велосипед»
**Причина:** код дублирует доступную библиотеку/util проекта.
**Фикс:** замените на библиотеку/util из каталога `docs/system-analysis/scan/reuse.json`.
Если каталога нет — он создаётся в фазе grounding (project-grounder/system-analyst).

## D. Субагенты, контекст, обрывы стрима

### D1. Фаза выполнилась inline (а должна — субагентом)
**Симптом:** оркестратор сам писал код/артефакты вместо `agent()`; `preflight-validate.py` → exit 1.
**Причина:** субагентная фаза (Design/Build/Verify/Document) сделана inline → теряется изоляция.
**Фикс:** фазы выполняются ТОЛЬКО через `agent(subagent_type="general-purpose", ...)`. Если так
случилось — шаг помечается, не продолжайте, пока не переделано субагентом.

### D2. `agent` недоступен в рантайме
**Симптом:** субагент не стартует (tool error).
**Действие:** конвейер выполнит фазу inline как **деградацию** (`degraded: true` в state) и явно
это пометит, чекпойнтя каждый микрошаг. Это допустимо как фоллбэк, но устойчивость ниже.

### D3. Частые обрывы стрима на тяжёлых фазах
**Причина:** раздут главный контекст (gradle/JaCoCo/сканы в основном агенте).
**Фикс:** тяжёлый вывод — в субагентах; между фазами — `read.py --excerpt-of`, не таскайте полные
выводы; перед тяжёлой фазой сожмите контекст. После обрыва — resume (см. user-guide §7).

### D4. Оркестратор «завис» / зациклился на вопросе
**Причина (дефект #7):** `agent()` и `ask_user_question` оказались активны одновременно → ответ теряется.
**Фикс:** вопросы — строго **до** запуска субагента или **после** его результата. Вопросы субагента
приходят через `pending_questions`, оркестратор их задаёт и перезапускает субагента с `answers`.

### D5. Вопрос не отобразился / пустой ответ
**Фикс:** повтор не более 1 раза, затем текстовое сообщение и fallback (`skipped: true` для шага, где можно).
Не давите третий раз.

## E. Jira / Bitbucket (MCP)

### E1. MCP не подключён
**Симптом:** нет инструментов вида `*jira*` / `*bitbucket*`.
**Действие:** пайплайн идёт в режиме «без Jira / до коммита». Для фаз 2.5 и 6 подключите MCP-серверы.
**Важно:** имена инструментов отличаются у серверов — **не угадывайте**, смотрите список доступных.

### E2. `check_jira` FAIL: «паритет нарушен»
**Причина:** число задач в Jira не совпадает с `task-plan.json` (например, создали 1 вместо 4).
**Фикс (рекомендуемый):** вернитесь в `tech-design`, поправьте `task-plan.json` до нужной разбивки,
пересоберите Jira-черновик — тогда сойдётся и downstream TDD/Build, и `check_jira`.
Осознанное укрупнение — только через override (§C3) с обоснованием.

### E3. Вопрос про Epic «потерялся»
**Причина:** `jira-task-writer` запущен inline вместо субагента → цикл `pending_questions` не отработал.
**Фикс:** фаза 2.5 — строго через `agent()`; вопросы (Epic/спринт) задаёт оркестратор по `pending_questions`.

## F. Тесты / покрытие / сборка

### F1. `coverage-judge` / `check_coverage` FAIL
**Причина:** покрытие изменённых файлов ниже порога (`quality.coverage_threshold`).
**Фикс:** конвейер вернёт тестописателя на добор (лимит 3). Если покрытие недостижимо без
инфраструктуры — зафиксируйте как ограничение, **не выдумывайте фейковые тесты**.

### F2. `initializationError` в тестах
**Причина:** `@DataJpaTest`/`@SpringBootTest` при стратегии `service-unit`.
**Фикс:** service-unit + моки; либо `quality.test_layer=mixed`. См. B2.

### F3. spotless/checkstyle падает
**Фикс:** конвейер **не запускает** `spotlessApply` молча — предложит вам. Запускайте форматтер сами,
затем новый коммит.

## G. State / манифест

### G1. Шаг не закрывается, хотя «всё сделано»
**Причина:** не выполнен `required_judges` шага (например, `brd-judge` для `00-brd`), или нет
`--artifacts` для шагов с файлами (`02-design`, `02-eval-plan`, `03-jira`).
**Фикс:** прогоните недостающего судью; закрывайте `update.py ... --status completed --artifacts '{...}'`.

### G2. Шаги `04-*`/`07-deliver-*` не сопоставляются с задачами
**Причина:** перепутан регистр task-id (`t1` вместо `T1`), или использован `add_steps.py` из
`pipeline-state/scripts/` (он не проставляет `required_judges`).
**Фикс:** используйте `feature-pipeline/scripts/add_steps.py`; id шагов — ровно как task-id (`T1`→`04-test-T1`).

### G3. `phase-gate` блокирует завершение (Stop)
**Причина:** остался шаг в статусе `in_progress`.
**Фикс:** закройте/пометьте `failed` (`update.py --status failed --error ...`) — не оставляйте висящим.

## Наблюдаемость (что смотреть при разборе)

```bash
# живой лог прогона (отдельный терминал)
bash <project>/.gigacode/hooks/watch-agents.sh

# логи субагентов по фиче
ls <project>/ground/ai-logs/<slug>/iter-NN/

# метрики из аудита
python3 <project>/.gigacode/hooks/agentops.py --root <project>
```

> Если после всех шагов проблема не воспроизводится через `preflight`/`doctor` — это, скорее всего,
> запуск без `--experimental-hooks`. Перепроверьте флаг первым делом.
